# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

本文件是仓库内 AI agent 的唯一项目级约束入口。`AGENTS.md` 只负责引导 Codex 读取本文件，不再重复维护同一套规则。

## 这是什么仓库

「喂今天」是一档给 9–15 岁孩子的**每日新闻播客**的产品/内容仓库。当前除 `新闻采集/` 外，主线仍以手工文档、稿子和音频验证为主；新闻采集是唯一获准独立工程化的模块。真实内容产出仍通过“出稿 → 看孩子反应 → 改 prompt → 次日再看”的高频循环逼近“什么是好稿”。

因此：不要给主线其它环节搭 App、自动推荐、自动写稿或自动发布脚手架。只有 `新闻采集/` 可以按其 README 和实施方案建设独立 Python CLI、测试与数据迁移；其它“开发”仍主要是写文档、写/改稿和迭代 prompt。

## Python 与依赖管理

`新闻采集/` 及仓库后续获准工程化的 Python 模块统一使用 **uv** 管理 Python 版本、虚拟环境、依赖和命令执行：

- 以 `pyproject.toml` 和 `uv.lock` 作为依赖事实源，必须提交二者；
- 使用 `uv python pin 3.12` 固定 Python 3.12；
- 使用 `uv add <package>`、`uv remove <package>` 修改依赖，不直接编辑锁文件；
- 使用 `uv sync` 创建或同步环境；
- 使用 `uv run <command>` 执行 CLI、测试、迁移和脚本；
- 不使用 `pip install`、`python -m venv`、Poetry、Pipenv 或 Conda 建立第二套项目环境；
- 除非排查 uv 自身问题，不直接调用 `.venv` 中的 Python 或工具。

## 文档结构与依赖关系

三份文档不是平级的，有明确的上下游：

- **`PRD_v0.9.md`** — 唯一的真理来源（source of truth）。定义了产品的魂、引擎五层和当前“只工程化新闻采集”的 MVP 边界。任何其它文档和任何稿子都不得与它冲突。改动产品决策**只在这里改**。
- **`Content_Safety_Policy_v0.1.md`** — PRD 第七节的执行细则。把"红线 / 敏感但重要 / 默认可聊"三层和家长三档开关填实到能用。明确声明**不动主线、不改任何已定设定、不给家长加新开关**。判定逻辑恒为 `家长档 × 年龄档`。
- **`Writing_Kit.md`** — 把 PRD 的魂压缩成一份**可直接粘进新对话当开场白**的写稿 prompt。每天用法："话题：[新闻]；档：[小学/初中]"。这是 PRD 第三节那套机制如何落进 prompt 的当前快照，也是 Narrative 引擎的雏形。
- **`正式文档索引.md`** — 正式文档导航和会话收尾检查表。它不定义产品决策，只登记当前运行指针、各文档职责、改动后的下游同步关系。每次改 prompt、手工流程、TTS 脚本或正式文档后，结束前要看它。

迭代方向：用 MVP 真实运行中的信号（**家长否决** + **AI 标"不确定"**）回流校准 → 校准 `Content_Safety_Policy` 的原则/红线，沉淀 `Writing_Kit` 的 prompt 规则。

## 当前在跑什么（MVP 运行态）

- **档位已锁定「小学高年级」**先跑通（用户 2026-06-29 决定）。初中档暂不写。
- **`prompts/`** — 正在反复迭代的写稿 prompt，按档位+版本命名（当前：`小学高年级_写稿提示词_v0.14.md`）。这是 `Writing_Kit.md` 收窄到单档、可迭代的运行版，**是当前出稿的真正依据**。每改一次升小版本、在文件内「迭代记录」写清改了什么和为什么。改 prompt 是这阶段最重要的工作（方法论第六步）。
- **`稿子/`** — 每天产出的单期文稿，命名 `日期_话题_档.md`，含元数据 + 选题记录 + 出稿自检 + 风险提示。
- **`手工跑通-嗨今天/`** — 「喂今天」主线尚未工程化、但已经进入重复验证的手工任务目录。每个编号子目录就是一个可执行任务，含目标、输入、步骤、产物、通过标准和沉淀位置；音频导演、TTS 生成、评估复听等过程文档放这里，不散落到项目根目录或 `探索/`。
- **`奥特曼咨询番外/`** — 与主线平级的定制资讯番外手工流程。只在家长手动触发时扫描奥特曼制作侧动态和可核验热点；有合格材料才给选题，无成刊不凑音频。它不改主线默认选题和 prompt。
- **`新闻采集/`** — 当前唯一独立工程化模块。服务器外部调度器定时触发一次性 CLI 任务，Supabase 为唯一运行库；不做推荐、家长选题、写稿或音频。
- **每日主线**：① 服务器定时运行 `新闻采集/` 得到新闻素材库 → ② 手工出 3 候选、标 1 推荐，家长选 1（PRD §7）→ ③ 按 `prompts/` 当前版本写整稿、自检、主动标风险 → ④ 如需音频，继续按 `手工跑通-嗨今天/` 验证。除新闻采集外，不搭 App、自动推荐、自动写稿、自动发布或分档系统。
- `Product_Methodology.md` — 「意义 → 机制 → 手动验证 → 才是系统」的打法；新闻采集进入工程化验证，其余环节仍处在手动验证。

## 写任何稿子或内容时必须守的硬约束

这些是 PRD「产品宪法」，违反任何一条都会让产品的核心机制失效——比文风偏好重要得多：

1. **教育必须隐身。** 禁止"今天告诉我们……""所以我们要……""小朋友应该……"。绝不说教、不下结论。
2. **问题比答案更重要。** 提出值得想的开放问题、不收口；不提供标准答案。
3. **爸爸不是老师，是和孩子一起探索的人。** 给 Narrative 的硬尺子：**爸爸的台词里，问句要多于结论。**
4. **首先要有趣。** 孩子忍着听完的稿子，再有营养也是失败的。
5. **北极星（思维方式/性格的那些名词）永远不许出现在稿子里，也不许照着它写。** 它只用于自检稿子够不够深，必须藏在五条机制里自然发生。
6. **一期 = 一条主题新闻**深聊 10–15 分钟，不是多条速览。
7. **AI 不决定孩子听什么，家长决定。** AI 只产出 3 个候选（标 1 个推荐）+ 理由，拍板的永远是人。

写稿的体例、长度、年龄分档讲法、自检清单，以 `Writing_Kit.md` 第五~七节为准。

## 编辑文档时的约定

- 文档带版本号与日期（如 `PRD_v0.9`、`v0.1`），并在文件顶部有「更新记录/版本」说明。做实质性修订时，更新版本号、日期，并在更新记录里写清"改了什么、为什么"——参照 PRD 更新记录的写法。
- 术语保持一致：核心机制、产品宪法、北极星、红线/敏感/默认三层、家长档 × 年龄档、引擎五层（Content Intelligence / Cognitive / Narrative / Family Personalization / Voice Performance）、常青稿、单主题。
- 区分"已定设定"（不轻易动）与「开放问题」（PRD 第十五节列出的待决项，如副标题、是否拆到 3 档、速览尾巴）。改前者要谨慎并说明理由。
- 全文中文。正式名称「喂今天」；「喂，今天」只作为历史代号或口播停顿写法保留，不作为平台正式名。
- 每次会话结束前，如果本轮改了 prompt、手工流程、TTS 行为、正式文档或沉淀规则，按 `正式文档索引.md` 的“会话结束文档检查”扫一遍，确认是否要同步当前指针、归档旧版或新增正式文档。
