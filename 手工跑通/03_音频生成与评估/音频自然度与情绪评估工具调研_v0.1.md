# 音频自然度与情绪评估工具调研 v0.1

**日期**：2026-07-04  
**状态**：MVP 手工跑通阶段调研  
**适用范围**：「喂，今天」小学高年级档，TTS 播客音频生成后的自动预检、自动小修与人工复听辅助

---

## 1. 结论

当前没有一个工具能可靠替代真人判断“这期播客自然、有对话感、孩子愿意听”。但自动评估可以进入生成闭环，先自动修掉可识别的小 bug，再把剩下的主观问题交给人抽听。

这里说的“情绪识别”不是让 LLM 听完以后猜，而是使用明确的 speech emotion recognition / speech prosody 模型输出标签、分数或原始结果：

- Hume Speech Prosody：输出细粒度韵律情绪维度，覆盖 calmness、interest、joy、sadness、surprise、tiredness 等。
- SenseVoice / FunASR：做 ASR 的同时输出 speech emotion recognition 和 audio event detection，适合本地预检。
- emotion2vec+：输出 9 类情绪分类，模型卡列出 `angry / disgusted / fearful / happy / neutral / other / sad / surprised / unknown`。

LLM 只能做两件事：把这些模型输出映射到本项目的导演意图，或生成可读报告。LLM 不允许凭文本内容替代声学识别结果。

自动评估最适合做四件事：

1. 找明显坏样本：读错、漏读、断裂、拼接突兀、太机械。
2. 找情绪不一致样本：导演稿写了惊讶/开心/担心，但模型听成平静或别的情绪。
3. 触发自动小修：改 pronunciation_dict、拆句、减标签、调停顿、调音量、重生成局部 turn。
4. 排序复听优先级：先听自动修不掉、分歧大、影响开头/结尾的句子。

建议当前阶段采用：

- **P0 本地评估**：`FunASR/SenseVoice` 或 `emotion2vec+` 做情绪粗分类；`SpeechMOS/UTMOS` 或 `NISQA-TTS` 做自然度 MOS 预估。
- **P1 云端精评**：`Hume Speech Prosody` 做更细的情绪/语气曲线分析，适合少量样本验证，不建议一开始全量依赖。
- **P0 自动小修**：每个可识别 bug 最多自动迭代 2-3 次，只改音频导演层，不改新闻事实和稿子主旨。
- **P0 人工抽听**：自动小修后，每期保留 6-10 个高风险 turn + 2 个随机 turn，由成人按固定表格打分。

不要把自动分数当“通过/不通过”的唯一依据。VoiceMOS Challenge 明确指出，合成语音的金标准仍是 human listening tests，客观指标存在和人类评分相关性不足、跨域泛化差的问题。工程上应把自动工具定位为“局部 bug 修复器 + 人工复听路由器”。

## 2. 工具分层

### 2.1 情绪/语气识别

| 工具 | 类型 | 能力 | 优点 | 限制 | 当前建议 |
|---|---|---|---|---|---|
| Hume Speech Prosody | 云端 API | 从说话方式识别细粒度情绪/语气 | 情绪维度细，覆盖 amusement、anger、calmness、confusion、interest、joy、sadness、surprise、tiredness 等 | 需要上传音频；成本和隐私要评估；标签细但未必等同播客导演意图 | 少量短样精评，验证情绪曲线 |
| FunASR / SenseVoice | 本地开源 | ASR、语言识别、情绪检测、音频事件检测 | 中文生态友好，可本地跑，适合批处理 | 情绪标签较粗；对 TTS 口播的“对话感”判断有限 | P0 首选本地预检 |
| emotion2vec+ | 本地开源模型 | 语音情绪识别，类别含 angry/disgusted/fearful/happy/neutral/other/sad/surprised/unknown | 专门做 SER，模型和论文公开 | 仍是粗粒度分类；中文播客场景要校准 | 可作为 SenseVoice 的第二意见 |
| openSMILE | 本地特征工具 | 提取 pitch、loudness、MFCC、eGeMAPS 等声学特征 | 适合诊断“为什么听着怪” | 本身不是成品评分器，需要后处理 | P1 诊断用 |

### 2.2 自然度/音质 MOS

| 工具 | 类型 | 能力 | 优点 | 限制 | 当前建议 |
|---|---|---|---|---|---|
| SpeechMOS / UTMOS | 本地开源 | 预测自然度 MOS | 接入简单，可批量跑；适合对比不同生成参数 | 分数要按本项目样本校准；不能判断内容是否像播客 | P0 批量排序 |
| NISQA / NISQA-TTS | 本地开源 | 预测 speech quality 和 TTS naturalness | 可输出 noisiness、coloration、discontinuity、loudness 等维度 | 更偏质量/自然度，不懂“爸爸像不像老师” | P0/P1 音质和断裂排查 |
| ASR CER/WER | 本地或云端 | 识别文本与原稿比对 | 抓错读、漏读、专名读错最有效 | 不直接代表自然度 | P0 必做 |

## 3. 推荐评估流程

### 3.0 情绪识别记录必须落盘

每一次识别都要保留原始记录，方便人工审计“模型是不是识别错了”。不要只保存一个 `emotion=happy`，必须保存模型名、版本、输入音频、原始输出、top-k、阈值和触发的修复动作。

建议产物：

```text
音频/xxx.emotion_raw.jsonl
音频/xxx.review_round1.csv
音频/xxx.fix_plan_round1.json
```

`emotion_raw.jsonl` 一行一个 turn：

```json
{
  "run_id": "2026-07-04T21-30-00_minimax_round1",
  "line_id": 12,
  "speaker": "孩子",
  "audio_path": "音频/xxx.minimax/part_012.mp3",
  "tts_text": "哇……这棵树，自己就是一片森林？！",
  "planned_cues": ["惊呼"],
  "expected_emotion_group": ["surprised", "interest"],
  "detectors": [
    {
      "name": "emotion2vec_plus_large",
      "version": "record exact model revision if available",
      "input_sample_rate": 16000,
      "output_type": "classification",
      "top_k": [
        {"label": "surprised", "score": 0.62},
        {"label": "happy", "score": 0.18},
        {"label": "other", "score": 0.09}
      ],
      "raw_output": {}
    },
    {
      "name": "sensevoice",
      "version": "record exact model revision if available",
      "output_type": "tag_or_classification",
      "top_k": [
        {"label": "surprised", "score": null}
      ],
      "raw_output": "<|zh|><|SURPRISED|><|Speech|>哇，这棵树自己就是一片森林"
    }
  ],
  "decision": {
    "status": "pass",
    "reason": "expected emotion appears in detector top_k",
    "auto_fix_allowed": false
  }
}
```

记录原则：

- 保存 `raw_output`，不要只保存清洗后的结论。
- 保存 `model_name/version/revision`，否则后续无法复现。
- 保存 `planned_cues` 和 `expected_emotion_group`，方便看“识别错”还是“导演稿写错”。
- 若模型只给标签不给概率，`score=null`，不要伪造置信度。
- 若多个模型冲突，状态写 `needs_review`，不自动修。
- 自动修复只在“模型输出和导演意图稳定错配”时触发；单模型低置信度不得触发强修复。

### 3.1 每句级预检

对 `line_concat` 路线，最好按每个 turn 的临时音频片段评估，而不是只评整期 mp3。

每句记录：

- `line_id`
- `speaker`：爸爸 / 孩子
- `planned_cue`：导演稿中的舞台提示，如 `（惊呼）`
- `tts_text`：实际发给 TTS 的文本
- `asr_text`：ASR 回读结果
- `cer`：字符错误率
- `mos_utmos` 或 `mos_nisqa`
- `emotion_top1/top3`
- `duration`
- `flags`

初始 flag 规则：

- `CER > 0.10`：疑似读错、漏读、专名错。
- `MOS < 本期中位数 - 0.4`：疑似自然度明显偏低。
- `duration` 明显长于同等字数句子：可能停顿过度或读法拖。
- `planned_cue` 是 `开心/惊讶/担心/难过`，但情绪识别 top3 完全不含相近类别：标为“情绪错配”，只提醒，不自动判死刑。
- 拼接后相邻 turn 静音超过 `1.2s` 且原稿没有 `<#x#>`：标为“疑似断裂”。

### 3.2 整期级预检

整期只看趋势，不看单一分数：

- ASR 回读是否能基本还原稿子。
- 爸爸/孩子音量是否明显不一致。
- 换人停顿是否稳定。
- 情绪曲线是否过平或过躁。
- 低分句子是否集中在某个 voice_id 或某类提示词。

### 3.3 自动迭代修复

自动修复只处理“局部、可识别、可回滚”的小 bug。不要让修复器重写新闻事实、重写爸爸/孩子关系、补教育总结。

闭环：

```text
导演稿 -> 逐句生成 -> 自动评估 -> 生成 fix_plan -> 局部改导演稿/参数 -> 只重生成失败 turn -> 再评估 -> 合成整期 -> 人抽听
```

建议最多迭代 2 轮；同一句第 3 次仍失败，标记为 `needs_human`，不要继续自动追分。

情绪相关修复的触发条件要更保守：

- `planned_cue` 明确存在，如 `惊呼/开心/紧张/难过/平静`。
- 至少一个声学情绪模型给出可审计输出，并已写入 `emotion_raw.jsonl`。
- 如果只有单模型输出，必须是高置信错配才自动修；低置信或 `unknown/other` 只进 review。
- 如果两个模型输出冲突，默认不自动修，只标 `emotion_disagreement`。
- LLM 根据台词推断出的情绪不能作为触发依据，只能作为报告里的参考说明。

| 可识别问题 | 识别信号 | 自动修复动作 | 停止条件 |
|---|---|---|---|
| 专名/术语读错 | ASR 和 `tts_text` 差异集中在实体词；CER 高 | 加 `pronunciation_dict`；把英文/专名改成带读音的写法；必要时拆短句 | ASR 回读接近原文 |
| 漏读/吞字 | ASR 缺少连续 3 个以上关键字 | 拆句；减少同句内标签；降低 speed | 关键字回读完整 |
| 停顿太长/太碎 | 静音检测或 duration 异常；多个 `<#x#>` 密集 | 删除多余 `<#x#>`；把 `<#0.80#>` 降到 `<#0.45#>`；合并过短句 | 时长回到同类句正常范围 |
| 拼接断裂 | 相邻 turn 静音超过阈值，且原稿没有停顿意图 | 降 `MINIMAX_PAUSE_SWITCH_MS`；局部合并短 turn；去掉句尾多余停顿 | 换人不突兀 |
| 情绪不足 | planned cue 是惊讶/开心/担心，识别为 neutral/calm/other | 优先改标点和口语词；再加 `emotion`；最后少量加 native tag，如 `(gasps)` | 情绪 top3 含相近类，且不过戏 |
| 情绪过头 | 识别为 angry/fearful/sad，但 planned cue 只是平静/温和 | 去掉 emotion；去掉感叹号和强 native tag；降低 vol/pitch/speed | 情绪回到 calm/neutral/other 附近 |
| 音质/自然度低 | MOS 明显低于本期中位数 | 去掉堆叠标签；减少 voice_modify；换同角色备用 voice_id；重生成该 turn | MOS 回到本期正常区间 |
| 爸爸/孩子音量不平 | RMS/LUFS 或听感统计差异大 | 每角色调 `vol` 或后处理 loudness normalize | 两角色响度接近 |

修复优先级：

1. 先修可懂度：错读、漏读、吞字。
2. 再修拼接：停顿、断裂、音量。
3. 再修情绪：不足或过头。
4. 最后修自然度：MOS 低但没有明确错误时，不要为了分数大改。

### 3.4 自动修复器输出格式

每轮生成一个可审计的修复计划：

```json
{
  "line_id": 12,
  "speaker": "孩子",
  "problem": "emotion_under",
  "evidence": {
    "planned_cue": "惊呼",
    "emotion_top3": ["neutral", "other", "calm"],
    "mos": 3.62,
    "cer": 0.01
  },
  "fix": {
    "action": "rewrite_delivery_only",
    "before": "哇，这棵树自己就是一片森林？",
    "after": "（惊呼）哇……这棵树，自己就是一片森林？！"
  },
  "retry": 1
}
```

修复计划必须落盘。人工回看时要知道系统为什么改、改了什么、改了几次。

### 3.5 人工复听表

自动工具筛出高风险句后，成人只听这些句子，按 1-5 分：

| 维度 | 问题 |
|---|---|
| 听得清 | 有没有错读、吞字、怪重音 |
| 像真人 | 有没有机械、拉扯、断裂 |
| 像对话 | 接上一句是否自然 |
| 情绪合适 | 有情绪但不过戏 |
| 角色稳定 | 爸爸像爸爸，孩子像孩子 |
| 可给孩子听 | 这一句是否会让孩子出戏 |

这比只问“自然吗”更稳定。

## 4. 和当前 MiniMax 路线的结合

MiniMax 当前先走 `line_concat` 单句生成拼接，因此评估天然可以句级化：

1. 生成时保留每句临时音频、请求 JSON、实际文本。
2. 对每句跑 ASR + 情绪识别 + MOS。
3. 生成 `音频/xxx.review.csv` 和 `音频/xxx.review.md`。
4. 生成 `音频/xxx.fix_plan.json`。
5. 自动修改导演稿副本或本句请求参数。
6. 只重生成失败 turn，不整期重跑。
7. 再次评估失败 turn，通过后重新 concat。
8. 只重听自动修不掉和抽样句子。

问题回写规则：

- `CER 高`：改 pronunciation_dict、改字词读法、拆句。
- `MOS 低`：减少标签、降低 emotion、换 voice_id 或重生成该 turn。
- `情绪错配`：优先改台词和标点，再少量改 emotion；不要每句堆标签。
- `拼接断裂`：调 `MINIMAX_PAUSE_SAME_MS / MINIMAX_PAUSE_SWITCH_MS` 或合并短句。
- `音量不稳`：优先做每角色 loudness normalize，再考虑改 `vol`。

建议产物命名：

```text
音频/xxx.review_round1.csv
音频/xxx.fix_plan_round1.json
稿子/xxx_minimax导演稿_autofix_round1.md
音频/xxx.review_round2.csv
音频/xxx.final_review.md
```

## 5. 建议实施顺序

### Step 1：先做最小评估脚本

新增一个本地脚本即可：

```text
tts/review_audio.py
```

输入：

```text
音频/xxx.minimax-request.json
音频/xxx.minimax/part_001.mp3 ...
```

输出：

```text
音频/xxx.review.csv
音频/xxx.review.md
```

先接：

- ASR：FunASR/SenseVoice
- 情绪：SenseVoice 或 emotion2vec+
- MOS：SpeechMOS/UTMOS

### Step 2：校准阈值

拿 10-20 句自己听过的样本做小标注：

- 好：自然、可用。
- 中：可接受但不够好。
- 差：必须重生成。

然后看自动分数和人工判断的关系。阈值只在本项目内生效，不要照搬论文或模型默认解释。

### Step 3：少量引入 Hume

只把每期最关键的 6-10 句发给 Hume Speech Prosody：

- 开头 hook
- 孩子惊讶/追问句
- 爸爸压低/留问题句
- 转折段
- 结尾开放问题

看它的情绪曲线是否和导演稿一致。Hume 更适合回答“这句听起来像 calm / interest / surprise / tiredness 吗”，不适合直接回答“这期播客孩子爱不爱听”。
